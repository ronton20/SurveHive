# Verifying SurveHive changes

Unity 6000.5.2f1 project; editor binary:
`/Applications/Unity/Hub/Editor/6000.5.2f1/Unity.app/Contents/MacOS/Unity`

## Runner script (use this instead of hand-typing Unity commands)

`scripts/unity.sh` wraps every batch invocation with the lock/zombie guard and
log parsing baked in. Run its batch subcommands with the harness
`run_in_background: true` (a pass is 1â€“3 min and can hang â€” never foreground-`timeout` it).

```
scripts/unity.sh lock                 # report editor/lock state
scripts/unity.sh build <Builder>      # SurveHive.BuildTools.<Builder>.Apply  (batch)
scripts/unity.sh validate             # BeehiveSceneValidator + prints verdict (batch)
scripts/unity.sh test EditMode        # -runTests, parses results XML (batch)
scripts/unity.sh test PlayMode        # boots the real Beehive scene + plays it
scripts/unity.sh drive [Method]       # visual capture, NO -batchmode, ~1 min GUI
```

Logs land in `VerifyShots/logs/` (gitignored). The `build`/`validate`/`test`/
`drive` subcommands refuse to run while the user's GUI editor is open (they need
the lock); `lock` tells you the current state. The raw commands each wraps are
documented below in case you need to vary them.

The guard distinguishes the user's GUI editor (`-useHub` in its args â€” never
kill it) from a stray headless run of mine (`-batchmode`/`-runTests`). If a
batch pass hangs and holds the lock: `kill <that-pid>`, `pkill -f
AssetImportWorker`, `rm -f Temp/UnityLockfile`, retry.

## Layers (run in this order)

1. **Builder pass(es)** â€” scene/assets are generated by idempotent editor
   passes; re-run the relevant one after changing builder code:
   `Unity -batchmode -quit -projectPath . -executeMethod SurveHive.BuildTools.<Builder>.Apply -logFile build.log`
2. **Scene validator** (static wiring checks, ~200 asserts):
   `... -executeMethod SurveHive.BuildTools.BeehiveSceneValidator.Validate -logFile validate.log`
   Grep the log for `[FAIL]` and `validation PASSED`.
3. **Tests**: `Unity -runTests -batchmode -projectPath . -testPlatform <EditMode|PlayMode> -testResults r.xml -logFile t.log`
   (PlayMode smoke test boots the real Beehive scene and plays it.)
4. **Visual drive with screenshots** (the real surface):
   `Unity -projectPath . -executeMethod SurveHive.BuildTools.PlayModeVerifyDriver.Run -logFile drive.log`
   â€” **no `-batchmode`** (the game view must render; opens the editor GUI
   for ~1 min). It auto-plays the Beehive scene, force-equips the active
   skills, triggers a level-up offer, saves screenshots to `VerifyShots/`
   (gitignored), and quits. Edit its staged `OnEditorUpdate` switch to
   drive whatever the change under test needs.

## Gotchas

- Exit code 2 from `-runTests` = test failures; parse the results XML.
- Game-time waits in PlayMode tests hang if the game pauses
  (`GamePause` sets `timeScale = 0` on level-up/death) â€” advance with
  unscaled-time caps and click through level-up offers (see
  `BeehiveSmokeTest.RunGameSeconds`).
- Screenshots while paused capture a stale framebuffer unless the driver
  calls `EditorApplication.QueuePlayerLoopUpdate()` every editor update.
- TMP components created in code default to NoWrap (project TMP settings)
  â€” long UI text needs `textWrappingMode = TextWrappingModes.Normal`.
- Unity batch runs take 1â€“3 min each; run them in the background.
