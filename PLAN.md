# SurveHive — Implementation Plan

The phased, **do-one-thing-at-a-time** execution plan for the open backlog in `TODO.md`,
ordered by importance and dependency. Each **sub-phase** is a self-contained unit of work you
can pick up, ship, and verify on its own. Do them top-to-bottom within a phase; phases are
also ordered, but B/C-tier phases can be interleaved where noted.

- `TODO.md` — the backlog/wishlist and rationale (why each item exists).
- `README.md` — living design doc; **update it in the same session** as any scope/mechanics change.
- `CLAUDE.md` — coding standards (mobile zero-GC, encapsulation, SOLID). All new code obeys these.
- `CHANGELOG.md` — release history.

**Legend:** ☐ not started · ◐ in progress · ✅ done. Touch-points list the primary files each
sub-phase edits/adds; they are a starting map, not an exhaustive diff.

> **Builder caution:** the Beehive scene + data assets were generated by the ordered
> `Assets/Editor/BuildTools/` passes, but they have since been **hand-tuned**. Do **not** re-run
> `BeehiveSceneBuilder.Build()` (or earlier phase builders) casually — it clobbers tuned data.
> Extend via new, additive, idempotent builder passes or targeted edits, and keep the
> validator green.

## Phase 1 — Combat System Overhaul 2.0 (the three power-up lanes)

The headline change: split the single flat skill pool into **three lanes** with per-lane
selection caps and per-card banners. This is the biggest richness-per-effort win and everything
else in combat hangs off it, so it goes first. Do the sub-phases **in order** — later ones lean
on the taxonomy the earlier ones establish.

**Target end-state of the lanes:**

| Lane | What it modifies | Distinct cap | Scales with |
|---|---|---|---|
| **Passive** | the player itself (never attacks) | **5** | its own level |
| **Enhancement** | the basic auto-attack | **3** | attack power + level |
| **Ability** | active auto-firing skills (separate from basic attack) | **5** | ability power + level |

Once a lane is at its cap, **no new** pick from that lane is offered — but already-owned picks
keep being offered to level up. Every offer card shows a **lane banner** and an **element** cue.

### 1A — Taxonomy: lane + element enums, card banners ✅
Foundation. Add the classification data and surface it on the card; no cap logic yet.
- Add `PowerUpLane { Passive, Enhancement, Ability }` and `SkillElement { Physical, Fire, Poison, Electric, Frost, Honey }` enums (append-only; enum indices are serialized in assets).
- Add `[SerializeField] _lane` and `_element` to `SkillDefinitionSO` (and expose read-only props). Ability cards infer/echo lane = Ability.
- Card UI: add a lane **banner/label** slot and an element-colored frame/gem to each choice card; drive them in `LevelUpUIController.ShowChoices()`. Wire placeholder art from `Assets/ThirdParty/FantasyStatusIcons/` (slice the sheet first — see 0-note below).
- Backfill `_lane`/`_element` on every existing `Assets/Data/Skills/*` asset.
- **Touch:** `Assets/Scripts/Progression/` (new enums), `Data/SkillDefinitionSO.cs`, `UI/LevelUpUIController.cs`, the level-up card prefab, `Data/Skills/*.asset`.
- **Done when:** every offered card visibly shows its lane banner + element tint; existing offers still work.

### 1B — Per-lane selection caps ✅
Gate the offer pool so each lane caps at its distinct count.
- Track owned-distinct count per lane (a skill is "owned" once its run-level ≥ 1). Config the caps (5 / 3 / 5) somewhere data-driven (constants or a small config SO).
- In `LevelUpUIController.BuildEligibleBuffer()`: an owned, non-maxed skill is always eligible (leveling); a **not-owned** skill is eligible only if its lane's owned count < that lane's cap.
- Edge cases: if all lanes are capped and everything eligible is maxed, keep the existing "nothing to offer → drain queue, resume" path. Ensure the auto per-level stat bump still applies.
- **Touch:** `UI/LevelUpUIController.cs`, `Progression/SkillOfferSelector.cs` (unchanged core, new eligibility feed), new cap-config constant/SO.
- **Tests:** EditMode — after 5 distinct passives owned, no 6th distinct passive is ever offered; owned passives still appear; same for 3 enhancements / 5 abilities.
- **Done when:** caps hold across a full run; tests green.

### 1C — Passive lane cleanup + new passives ✅
Reshape the Passive lane to the intended roster and add the two new stats.
- **Add `armor`** to `PlayerStats` — damage-taken reduction. Model as flat + %-mitigation applied at the player's damage intake (in `HealthComponent`/`PlayerHitFeedback` receive path). Add a `PassiveArmor` skill.
- **Add `ability power/damage`** to `PlayerStats` — a multiplier read by `ActiveSkillManager` when it computes ability damage (multiply `stats.Damage` at fire time). Add an `AbilityPower` skill.
- **Migrate out of Passive:** re-tag Projectile Count (Twin Stingers) and Attack Range (Longer Stinger) as **Enhancement** lane (they shape the basic attack) — they get folded into 1D. Keep pure-player stats here: move speed, max HP, armor, attack power, attack speed, ability cooldown, ability power, crit chance, crit damage, lifesteal, magnet.
- **Touch:** `Player/PlayerStats.cs`, `Health/HealthComponent.cs` (armor intake), `Combat/Skills/ActiveSkillManager.cs` (ability-power multiply), `Progression/SkillEffectType.cs` + `SkillEffectApplier.cs` (new effect types), `Data/Skills/*`.
- **Done when:** armor measurably reduces damage taken; ability power measurably raises Ability damage; the Passive lane holds exactly the player-stat roster.

### 1D — Enhancement lane (new): basic-attack modifier system ✅
The genuinely new system. A composable modifier layer on the basic auto-attack.
- New `PlayerAttackModifiers` component (or extend `PlayerStats`) holding basic-attack modifier state: bonus projectile count, per-projectile damage factor, pierce count, on-hit status procs (type/chance/potency/duration), bonus range, elemental tag.
- `AutoAttack.FireAtTarget` reads these: projectile count and per-projectile damage factor (Multishot: +1 projectile, each softer, ~1.5× total per level), range from modifiers.
- `Projectile` gains a **pierce count** (survives N hits before releasing) and applies **on-hit status** (ignite → burn) via `StatusEffectReceiver`, gated by proc chance.
- Enhancement roster: **Multishot** (projectile +1, damage-adjusted), **Piercing Shot** (−damage, +pierce — this is the retired active moved here), **Ignite** (on-hit burn), **Lifesteal** (attack lifesteal — note global lifesteal already exists in `DamageService`; decide whether this is an enhancement-only bump or reuse), **Long Sting** (attack range), plus room for more elemental procs (chill/shock/venom on-hit) as elements land.
- **Retire** the standalone `Piercing Lance` active skill (or repurpose its asset) so pierce lives in this lane. Update `README` active-skill count.
- **Touch:** new `Combat/PlayerAttackModifiers.cs`, `Combat/AutoAttack.cs`, `Combat/Projectile.cs`, `Progression/SkillEffectType.cs` + `SkillEffectApplier.cs`, `Data/Skills/*`, `Data/ActiveSkillSO` (retire Lance).
- **Tests:** EditMode — Multishot total-damage curve (~1.5× per level), pierce hit count, on-hit burn application.
- **Done when:** taking Multishot adds projectiles with the intended total-damage scaling; Piercing Shot pierces; Ignite burns; all three respect the 3-cap.

### 1E — Ability lane expansion ✅
Grow and polish the active-skill roster; wire ability-power scaling (from 1C).
- **Make the 360° stinger burst (RadialVolley / Stinger Barrage) pierce** — its projectiles pass through enemies.
- Add new abilities across behaviors/elements: **Honey Bomb** (lobbed AoE burst), **Chain Lightning** (extend/second ChainArc), **Exploding Stinger** (HomingBolt explode variant), **Poison Cloud** (Aura, if not already covered by Pollen Cloud) — reuse existing `ActiveSkillBehavior`s where possible; add a new behavior only if a delivery truly doesn't exist.
- Ensure `ActiveSkillManager` multiplies damage by ability power (1C) and respects the 5-cap via the lane gating (1B). `MaxEquipped` is already 8 — fine.
- **Touch:** `Combat/Skills/ActiveSkillManager.cs`, `Data/ActiveSkillSO.cs` (if new behavior), new `Data/Skills/*` ability assets + pool IDs, `Data/SkillDatabaseSO`.
- **Done when:** stinger burst pierces; ≥3 new abilities are offerable and scale with ability power; 5-cap holds.

### 1F — Power-up readability UX ✅
Help players strategize around the lane caps now that the lanes exist. Pure surfacing of state
the controller already tracks — no new mechanics. Cheap; slot after the functional lanes land.
- **Lane counter on the offer card** — under each card's lane banner, show an `owned/cap`
  counter (e.g. `1/5`, `2/3`, `4/5`) so the player sees at a glance how much room a lane still
  has before committing. `LaneEligibility` already computes owned-per-lane; expose those counts
  and render a small counter under the banner label (extend the `Combat 2.0/1A` banner pass).
- **Owned power-ups list in the pause menu** — a panel listing every power-up owned this run,
  grouped by lane, each with current level (and element cue). Reads the same per-run level
  state `LevelUpUIController` holds (`_skillLevels`); the pause menu already exists (Phase 4C).
- **Touch:** `UI/LevelUpUIController.cs`, `Editor/BuildTools/CombatOverhaulBuilder.cs` (counter UI), `UI/PauseMenuController.cs` + new owned-list panel, a run-scoped read of owned skills/levels.
- **Done when:** each offer card shows its lane's `owned/cap`; pausing shows the current build grouped by lane.

## Phase 2 — Boss & Wave Drama (feel)

Makes the run's set-pieces land. Independent of Phase 1 and cheap relative to payoff; can be
interleaved with late Phase 1 if desired.

### 2A — Pre-spawn warnings (5s lead) ✅
- 5 seconds before a strong wave or a boss/miniboss event fires, show a warning banner ("⚠ Royal Guard incoming…") counting in until the spawn.
- `StageDirector` currently fires events at their normalized time. Add a **lookahead**: convert 5s to normalized progress (`5 / TotalDurationSeconds`) and, per event, raise an `OnStageWarning(event)` when progress crosses `eventTime − lead`, then the real event at `eventTime`. Guard against double-firing (track warned events like `_firedBuffer`).
- New/updated warning-banner UI (reuse `BossBannerUI` styling or a lighter HUD toast).
- **Touch:** `Stage/StageDirector.cs`, `Stage/StageTimeline.cs` (warning-window collection), `UI/` new warning banner, HUD wiring.
- **Done when:** every strong wave and boss shows a 5s warning that resolves exactly as the spawn happens.

### 2B — Impactful miniboss kill: guaranteed lucky reward ☐
- On **miniboss** death (not the final boss), grant a **guaranteed lucky power-up** — a level-up offer where the picked card grants **+2 levels** (still a random offer set) — plus a burst of EXP.
- `BossSpawner.HandleGatingBossDied` already distinguishes `_gatingBossIsFinal`. For the non-final case, trigger: (a) award bonus EXP via `PlayerExperience`, and (b) force the next level-up offer to be lucky. Add a "force lucky next offer" hook to `LevelUpUIController` (or a queued flag on `PlayerExperience`) so `ShowChoices` marks the cards lucky.
- Sequence this **after** the 2C death animation completes (see ordering).
- **Touch:** `Stage/BossSpawner.cs`, `Progression/PlayerExperience.cs`, `UI/LevelUpUIController.cs`.
- **Done when:** killing the Royal Guard reliably yields a +2 lucky pick and an EXP pop.

### 2C — Boss/miniboss death sequence: slow-mo + invuln + shockwave ✅
- On any boss/miniboss death: enter **slow-motion** (scaled `Time.timeScale`, e.g. 0.25×) for the death animation duration, make the **player invulnerable** for that window, fire a **shockwave VFX + screen shake**, and **hold** all downstream events (miniboss reward 2B, timeline resume, victory) until the animation completes.
- Slow-mo must cooperate with `GamePause` (which owns `Time.timeScale`) and `HitStop` — add a `SlowMotion`/`BossDeathSequence` coordinator that yields to a real pause and restores `timeScale` cleanly. Because gameplay uses scaled time, a coroutine on unscaled time drives the sequence.
- Player invuln: a temporary flag on the player's damage intake (`HealthComponent`/`PlayerHitFeedback`) — reuse or extend `PlayerShield`'s ignore path.
- Boss death currently resumes the timeline immediately (`SetBossActive(false)`) / shows victory instantly — gate both behind the sequence's completion.
- **Touch:** new `Core/BossDeathSequence.cs` (+ maybe `Core/SlowMotion.cs`), `Core/GamePause.cs`/`HitStop` cooperation, `Stage/BossSpawner.cs`, `Enemies/QueenBossController.cs` + `EnemyController.cs` (death-anim length signal), player invuln path, shockwave VFX pool.
- **Done when:** every boss/miniboss death plays a dramatic slow-mo beat with shake + shockwave, the player can't die during it, and rewards/victory/resume only fire afterward.

## Phase 3 — Damage model depth

From `TODO.md` #20 → #23 → #19. Do in order; each is a prerequisite for the next. Builds on the
element tags from Phase 1A.

### 3A — Damage typing (physical vs magic) — TODO #20 ☐
- Every damage application carries a `DamageType { Physical, Magic }`. Auto-attack/stingers/enhancements = Physical; magic abilities (honey/pollen/ember/static) = Magic. `DamageService` and status DoTs stamp the type.
- **Touch:** `Combat/DamageService.cs`, `Health/IDamageable.cs`/`HealthComponent.cs`, `Combat/Status/*` DoT, all damage call-sites.

### 3B — Enemy defenses beyond HP — TODO #23 ☐
- Ordered mitigation pipeline on the enemy: **shield → armor → HP**, reading incoming `DamageType` (physical shield vs magic shield; armor reduces physical). Show state via tint/overlay bar.
- **Touch:** `Enemies/EnemyController.cs`, `Health/HealthComponent.cs`, `Data/EnemyStatsSO.cs`, `UI/EnemyHealthBarUI.cs`.

### 3C — Elemental set effects — TODO #19 ☐
- Owning 2 / 3 / 4+ enhancements+abilities of the same element grants escalating set bonuses. Data-driven `SetBonusSO` per element; track owned counts per element on the player; show active tiers on the HUD.
- **Touch:** new `Data/SetBonusSO.cs`, `Progression/` element-count tracking, HUD set-tier indicator.

## Phase 5 — Release hardening

The tail of the build push. Wants Phases 1–4 in place plus real playtest feedback.

### 5A — Difficulty / curve tuning pass ☐
- Balance EXP curve, enemy scaling, spawn curve, drop rates, boss HP against the new lanes/defenses. Target: first-time player dies **~minute 8–12**; a meta-invested player can **clear** the Queen. Document what changed and why.

### 5B — Mobile UI overhaul (TODO #14–17) + sanity pass (#24) ☐
- **Safe-area anchoring** (most urgent — the notch hides HUD meters) → scale pass for small screens → skill cards relayout to horizontal/stacked for portrait → boss bar/markers/results audit on 19.5:9+. Then verify joystick + tap flows in the Device Simulator on iPhone + a tall Android, confirm safe areas, texture memory, and a zero-GC / frame-rate profiling pass.

### 5C — Localization seam ☐
- Route all user-facing strings through one string table/asset instead of hardcoded literals (actual translation deferred). Cheaper before 5B touches all that text — do it just ahead of the mobile UI pass.
